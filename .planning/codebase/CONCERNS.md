## CRITICAL Risks

### 1. Firestore Security Rules Gap — Anonymous Users Can Read All Data
**Severity: CRITICAL**
Anonymous Firebase users can read all sensitive student, teacher, and financial data. `firestore.rules` acknowledges this at lines 55-59 as a planned v14.4 fix, but it has not been implemented. Any unauthenticated visitor with knowledge of the collection path can read the entire dataset.

**Impact:** PII exposure (student names, phone numbers, birthdates), financial data, instructor contact info.

**Fix needed:** Implement server-side Firestore security rules that enforce role-based reads before any data is returned.

---

### 2. Bulk-Overwrite Vulnerability on Non-Student Collections
**Severity: CRITICAL**
The `saveStudents([...])` hard-lock prevents the catastrophic array-overwrite pattern for students only. All other collections — `saveAttendance`, `savePayments`, `saveTeachers`, etc. — still use full array overwrite. The same race condition that caused the 77-student data loss incident can recur on any of these collections.

**Impact:** Silent data loss for attendance records, payment history, teacher data.

**Fix needed:** Apply the per-op transaction pattern (`addDoc`, `updateDoc`, `deleteDoc`, `runTransaction`) to all write operations across all collections.

---

### 3. Console Logs Leak Firebase UIDs in Production
**Severity: CRITICAL**
`src/aiClient.js` contains 3 `console.log` calls that output Firebase UIDs on every AI request in production. Any visitor with DevTools open can see user identifiers.

**Fix needed:** Remove all `console.log` calls from `src/aiClient.js`.

---

### 4. `resetSeed()` Reachable in Production AdminTools
**Severity: CRITICAL**
The sample data reset function (`resetSeed()` / "샘플 데이터 초기화") is accessible via AdminTools in the production build. Triggering it will wipe the live Firestore database and replace it with seed data. No confirmation guard exists (or exists only as `window.confirm` which was banned).

**Impact:** Complete production data loss.

**Fix needed:** Remove this function entirely from the production build, or gate it behind a build-time env flag (`import.meta.env.DEV`).

## HIGH Issues

### 5. Firebase Auth ↔ Local Session Desynchronized
**Severity: HIGH**
Firebase Auth and the local `localStorage` session are out of sync. `localStorage` session tokens never expire. The app is currently operating on a local fallback for auth, meaning password changes in Firebase Auth do not propagate correctly to active sessions.

**Impact:** Stale sessions persist indefinitely; compromised accounts cannot be force-logged-out.

---

### 6. `callAnthropic` Actually Calls Gemini
**Severity: HIGH**
`functions/api/ai/_utils/anthropic.js` is named `anthropic.js` and exports `callAnthropic`, but internally it calls the Gemini API. This naming mismatch is a maintenance trap — any developer (or Claude) will assume it calls Anthropic/Claude and may configure it incorrectly.

**Fix needed:** Rename to `gemini.js` / `callGemini`, or actually wire it to the Anthropic SDK as intended.

---

### 7. Rate Limiter Silently Disabled When KV Binding Missing
**Severity: HIGH**
The AI endpoint rate limiter silently no-ops when the Cloudflare KV binding is not configured. There is no error or warning — requests pass through unthrottled. In production without the binding, the AI endpoints are fully open to abuse.

**Fix needed:** Make the KV binding required; throw a startup error if it's absent.

---

### 8. `src/App.jsx` God-File (840+ lines)
**Severity: HIGH**
`App.jsx` exceeds the 800-line project limit and contains: router logic, all Firestore real-time listeners, global state management, auth handling, seed data generation, and one-time migration code. This makes it extremely difficult to maintain and reason about.

**Fix needed:** Extract listeners to a `useFirestoreListeners` hook, migration code to a separate utility, and seed data to a dev-only module.

---

### 9. Real PII Hardcoded in Seed Data (Committed to Public Repo)
**Severity: HIGH**
The `generateSeedData()` function in `App.jsx` contains real-looking Korean names, phone numbers, and birthdates. If any of this reflects real people (or if it was derived from real data), it is PII committed to a public GitHub repository.

**Fix needed:** Replace with obviously synthetic data (e.g., "홍길동", "010-0000-0000").

---

### 10. No Automated Test Suite
**Severity: HIGH**
There are zero automated tests. Validation is entirely manual: `npm run build` + browser verification. Regressions in business logic (attendance calculations, fee computations, role-based data filtering) cannot be caught automatically.

**Fix needed:** Add Vitest for unit tests covering `src/utils.js` helpers (attendance rate calculations, fee logic, institution expansion). Aim for 80% coverage on pure functions first.

## MEDIUM Issues

### 11. CSS as Inline JS String Constants — No Tooling
**Severity: MEDIUM**
All CSS is stored as string constants in `src/constants.jsx` and injected via `<style>` tags at runtime. There is no CSS scoping, no dead-code elimination, no linting, and no IDE autocomplete for CSS properties.

**Impact:** CSS bugs are invisible at build time; styles can conflict silently.

---

### 12. One-Time Migration Code Permanently in App.jsx
**Severity: MEDIUM**
Migration guards (e.g., `ryekPwResetV12Done`, `rye-recovery-v1`) and their associated migration logic live permanently in `App.jsx`, running checks on every app load. This is dead weight for all users who have already migrated.

**Fix needed:** Move completed migrations behind a build-time flag or remove them after the migration window has passed.

---

### 13. 880KB Single JS Bundle — No Code Splitting
**Severity: MEDIUM**
The entire app ships as one ~880KB JS bundle (~220KB gzip). There is no route-based code splitting, no vendor chunk separation, and no lazy loading of heavy views. Initial load time is penalized for all users.

**Fix needed:** Add `React.lazy()` + `Suspense` for view-level code splitting; configure Vite's `build.rollupOptions.output.manualChunks` to separate vendor code.

---

### 14. All Firestore Data in Single Documents Per Type
**Severity: MEDIUM**
All records of each type (students, teachers, attendance, etc.) are stored as arrays in single Firestore documents under `appData`. Firestore has a 1MB document size limit. As attendance records and activity logs accumulate, this will hit the limit and writes will silently fail.

**Fix needed:** Migrate to subcollections (one document per record) for high-volume collections (attendance, payments, activity logs).

---

## LOW Issues

### 15. `monthlyFee` is 0 for All Students
**Severity: LOW**
All student records have `monthlyFee: 0`. Payment analytics, revenue dashboards, and churn predictions that depend on fee data are operating on meaningless values.

**Impact:** Any AI-powered revenue analysis will return incorrect results.

---

### 16. Activity Log Silently Truncated at 200 Entries
**Severity: LOW**
The activity log is truncated to 200 entries without any notification to the user. Historical audit trails are lost silently.

---

### 17. AI Integration Partially Stubbed
**Severity: LOW**
The AI features (`functions/api/ai/churn.js`, `functions/api/ai/query.js`) and the new `src/components/ai/` directory and `src/components/dashboard/ChurnWidget.jsx` are partially implemented. It is unclear which endpoints are fully wired vs. stubbed with placeholder responses.
